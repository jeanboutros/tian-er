## 1. Overview

### 1.1 Purpose

C12 Service Orchestration is the **runtime wiring** of the Tian'er Signal Intelligence Platform. It defines every systemd `--user` unit generated by Quadlet from the declarative `.container`, `.pod`, `.volume`, and `.network` files [3] deployed by C14. C12 owns the dependency graph, the startup and shutdown ordering, the restart policies [2], the per-service security sandboxing (ReadWritePaths, NoNewPrivileges, SystemCallFilter, capability drops [5]), the timer-based PCAP rotation and batch deep-parse schedules, the target hierarchy, and the `tianer.target` top-level orchestration unit.

Where C14 builds images and deploys Quadlet files to the filesystem, C12 defines **what happens when systemd starts**. It is the difference between "the files are installed" and "the platform is running."

### 1.2 Scope

| In Scope | Out of Scope |
|----------|-------------|
| systemd `--user` dependency graph for all 11 container types | Container image builds (C14) |
| Quadlet `.container`, `.pod`, `.volume`, `.network` unit file specifications | Quadlet file deployment to `/etc/containers/systemd/` (C14) |
| `tianer.target` top-level orchestration unit | Host-level systemd units (C01: `systemd.unit` for tmpfiles, udev, linger) |
| Per-service security sandboxing: ReadWritePaths, NoNewPrivileges, SystemCallFilter, --cap-drop ALL | Application-level security logic (per-component own sections) |
| Restart policies (Restart=on-failure, RestartSec, StartLimitBurst) [2] | Container health check scripts (component-specific: C05 ping DB, C09 `/api/health`) |
| Timer units for oneshot containers (PCAP rotation, deep parser batch) | Timer script contents (C04, C07) |
| Startup and shutdown ordering (Before=/After=/Requires=/Wants=) | Per-component service readiness logic |
| Container-to-container communication path definitions (Contracts) | Database connection pooling (C02, C09) |
| `systemctl --user` commands for operator control | `setup.sh` phases (C14) |

### 1.4 Position in the System

C12 is **Layer 4** in the build sequence — the final component (component-breakdown.md §4.1). It depends on all other components being defined because it wires their Quadlet units together into a coherent dependency graph. It is the last component designed and the first one verified after deployment: if `systemctl --user start tianer.target` succeeds, the platform is running.

**Key architectural decisions:**

| Decision | Rationale |
|----------|-----------|
| Capture pod `Network=none` | Sniffers talk only to USB hardware and FIFOs. No network access eliminates attack surface. No containers in this pod require TCP connectivity — ingest bridge runs in the platform pod on `tianer-net`. |
| Platform pod on `tianer-net` bridge | API, gap detector, deep parser, and other platform services need TCP access to PostgreSQL and (in the future) to each other. All on a private bridge with no upstream route. |
| PostgreSQL + Grafana as standalone containers | These services have their own lifecycle and do not benefit from pod co-location. PostgreSQL persists data across pod restarts. Grafana may be restarted independently for plugin updates. |
| `--group-add keep-groups` on capture containers | Inherits `plugdev`, `dialout`, `wireshark` host groups for USB device access. |
| `--cap-drop ALL` default everywhere | Minimum privilege: only PostgreSQL gets `CHOWN DAC_OVERRIDE FOWNER SETGID SETUID`; sniffers get `CAP_NET_RAW` and `CAP_NET_ADMIN` [5]. All other containers drop every capability. |

### 2.3 Container Startup Ordering

The startup sequence follows a **waterfall dependency model** — each layer waits for the layer below to be ready:

```
Time ─────────────────────────────────────────────────────────────►

Phase 1: Network (0s)
  tianer-net.network ................. [CREATED]
  (bridge network must exist before any container on it starts)

Phase 2: Database (0–15s)
  tianer-postgres.service ............ [STARTING]
    ├─ podman run tianer-postgres .... [PULLING/STARTING]
    ├─ PostgreSQL init scripts ....... (first run only)
    ├─ WAL recovery .................. (crash recovery, if needed)
    └─ pg_isready .................... [READY] (~5-10s)
  tianer-postgres.service ............ [RUNNING]

Phase 3: Pods + Grafana (15–30s, parallel)  ← requires postgres
  tianer-capture-pod.service ......... [STARTING]
  tianer-platform-pod.service ........ [STARTING]
  tianer-grafana.service ............. [STARTING]
    │
    │  Within capture pod (per-sniffer instance, sequential):
    │    tianer-tshark@ut1.service .... [STARTING]
    │    tianer-sniffer@ut1.service ... [STARTING]
    │      ├─ Opens USB device ........ (udev symlink)
    │      └─ Begins capture .......... [CAPTURING]
    │    tianer-heartbeat@ut1.service . [PULSING]
    │    ... (repeat for nrf1, nrf2, nrf3 if enabled)
    │
    │  Within platform pod (parallel):
    │    tianer-ingest@ut1.service .... [STARTING]
    │      ├─ Waits for DB connection . (exponential backoff)
    │      └─ Pre-warms batch buffer .. [READY]
    │    ... (repeat for nrf1, nrf2, nrf3 if enabled)
    │    tianer-api.service ........... [STARTING]
    │      └─ FastAPI lifespan ........ (pool warm, health OK)
    │    tianer-gap-detect.service .... [STARTING]
    │    tianer-prometheus.service .... [STARTING]
    │    tianer-promtail.service ...... [STARTING]
    │
    │  tianer-grafana.service ........ [STARTING]
    │    └─ Datasource health check ... (verifies PG connection)

Phase 4: Ready (30s)                ← tianer.target reached
  systemctl --user is-active tianer.target → active
  All containers running and healthy.
```

**Total startup time target:** Under 30 seconds from `systemctl --user start tianer.target` to all containers ready (component-breakdown.md §12.4).

**Concurrency note:** Pod start is serial within each pod (Quadlet starts pod members in dependency order) but parallel across pods. PostgreSQL must be ready before any consumer starts — consumers implement exponential backoff reconnect (1s, 2s, 4s, 8s, cap 30s per C02 §6.2) if PostgreSQL is slow to accept connections.

### 2.4 Target Hierarchy

The systemd `--user` instance for the `tianer` user uses a three-level target hierarchy:

```
default.target (systemd --user default)
    │
    └── WantedBy=default.target
        │
        └── tianer.target (C12 top-level target)
            │
            ├── Wants=tianer-postgres.service
            ├── Wants=tianer-capture-pod.service
            ├── Wants=tianer-platform-pod.service
            └── Wants=tianer-grafana.service
```

`tianer.target` is the single unit that operators use to start/stop the entire platform:

```bash
# Start everything
systemctl --user start tianer.target

# Stop everything (gracefully shuts down PostgreSQL last)
systemctl --user stop tianer.target

# Restart everything
systemctl --user restart tianer.target

# Check status
systemctl --user status tianer.target
```

### 3.2 Template Unit Naming Convention

| Template | Instances (4-sniffer config) |
|----------|---------------------------|

Disabled sniffers in `sniffers.yaml` (where `enabled: false`) have their template units **not instantiated**. The C14 deployment logic in `setup.sh` Phase 11 scans `sniffers.yaml` and enables only units for enabled sniffers:

```bash
# In Phase 11 (services enable):
for sniffer in $(grep -A5 'enabled: true' /etc/tianer/sniffers.yaml | grep 'name:' | awk '{print $2}'); do
    systemctl --user enable tianer-sniffer@${sniffer}.service
    systemctl --user enable tianer-tshark@${sniffer}.service
    systemctl --user enable tianer-ingest@${sniffer}.service
    systemctl --user enable tianer-heartbeat@${sniffer}.service
done
```

### 3.3 Timer Units

Oneshot containers that run on a schedule use systemd `.timer` units:

| Timer | Triggers | Schedule | Purpose |
|-------|---------|----------|---------|
| `tianer-rotate.timer` | `tianer-rotate.service` | Every `BLESNIFF_ROTATION_MINUTES` (default 30 min) | PCAP file rotation + compression |
| `tianer-deep-parse.timer` | `tianer-deep-parse.service` | Hourly after `tianer-rotate.timer` completes | Batch PCAP deep parsing |

Timers are defined as part of their respective Quadlet `.container` files using the `[Timer]` section (Quadlet supports timer directives). These timers run under the `tianer` user's systemd `--user` instance.

**Timer Quadlet pattern (tianer-rotate.container):**

```ini
[Unit]
Description=Tian'er PCAP Rotation
After=tianer-postgres.service

[Container]
Image=localhost/tianer-rotate:latest
EnvironmentFile=/etc/tianer/tianer.env
EnvironmentFile=/etc/tianer/blesniff.env
Volume=/etc/tianer:/etc/tianer:ro
Volume=/var/lib/tianer/pcap:/var/lib/tianer/pcap:rw
Volume=/var/log/tianer:/var/log/tianer:rw
Pod=tianer-platform.pod
User=tianer
Group=tianer
NoNewPrivileges=true
DropCapability=ALL

[Service]
Type=oneshot
RemainAfterExit=yes

[Timer]
OnUnitActiveSec=${BLESNIFF_ROTATION_MINUTES}m
Persistent=true

[Install]
WantedBy=tianer.target
```

## 4. Low-Level Architecture (LLA)

### 4.1 Per-Service Quadlet Unit Specifications

This section specifies the complete Quadlet configuration for every service. Each specification includes `[Unit]` ordering, `[Container]` volume mounts and security directives, `[Service]` restart policies, and `[Install]` target integration. Volume IDs (V01–V09) reference the storage-strategy.md inventory.

#### 4.1.1 tianer-net (Bridge Network)

```ini
# deploy/containers/tianer-net.network
[Network]
Subnet=10.89.0.0/24
Internal=true
Label=app=tianer,component=network
```

**Properties:**
- `Internal=true`: No external gateway. Containers on this network can reach each other but cannot access the internet or host network.
- `Subnet=10.89.0.0/24`: Static subnet for predictable container IPs.

#### 4.1.9 tianer-sniffer@.container (Capture Pod — Template)

```ini
# deploy/containers/tianer-sniffer@.container
[Unit]
Description=Tian'er Sniffer %i Wrapper
After=tianer-tshark@%i.service
After=tianer-ingest@%i.service

[Container]
Image=localhost/tianer-sniffer:latest
ContainerName=tianer-sniffer-%i
Pod=tianer-capture.pod

# Environment
EnvironmentFile=/etc/tianer/tianer.env
EnvironmentFile=/etc/tianer/blesniff.env
Environment=SNIFFER_NAME=%i

# Volume mounts
Volume=/etc/tianer:/etc/tianer:ro                       # V01: config + sniffer.yaml
Volume=/var/lib/tianer/pcap:/var/lib/tianer/pcap:rw     # V02: PCAP output
Volume=/var/lib/tianer/heartbeat:/var/lib/tianer/heartbeat:rw  # heartbeat files
Volume=/var/run/tianer:/var/run/tianer:rw               # V03: FIFOs

AddDevice=/dev/tianer/%i

# Security hardening — sniffer needs elevated capabilities for raw capture
User=tianer
GroupAdd=keep-groups   # plugdev, dialout, wireshark for USB access
NoNewPrivileges=true
DropCapability=ALL
AddCapability=NET_RAW NET_ADMIN                       # dumpcap needs these
SystemCallFilter=@system-service @file-system @process @signal @io-event
ReadWritePaths=/var/lib/tianer/pcap /var/lib/tianer/heartbeat /var/run/tianer /var/log/tianer /tmp

# Health check — heartbeat file age
HealthCmd=test $(($(date +%s) - $(stat -c %Y /var/lib/tianer/heartbeat/%i.ts 2>/dev/null || echo 0))) -lt 120
HealthInterval=30s
HealthTimeout=5s
HealthStartPeriod=15s

# Resource limits
MemoryMax=256M
CPUQuota=50%

[Service]
Restart=on-failure
RestartSec=5s
StartLimitBurst=3
TimeoutStartSec=30

[Install]
WantedBy=tianer.target
```

**Key design decisions:**
- `AddDevice=/dev/tianer/%i`: The `%i` template instance (e.g., `ut1`, `nrf1`) matches the udev symlink name under `/dev/tianer/`. This is the canonical mechanism for USB device passthrough.
- `GroupAdd=keep-groups`: Inherits the `tianer` user's supplementary group memberships (`plugdev`, `dialout`, `wireshark`) from the host. Required for USB device access inside the container.
- `AddCapability=NET_RAW NET_ADMIN`: Only sniffer containers receive these. All other pods and standalone containers drop ALL capabilities.
- Health check verifies the heartbeat file age is under 120 seconds (significantly more generous than the 30s heartbeat interval, to avoid false-positive restarts during transient capture bursts).
- `MemoryMax=256M`: Conservative limit. The sniffer is a thin wrapper around `ubertooth-btle` or `nrfutil ble-sniffer` — it streams bytes to PCAP and the FIFO, doing no buffering itself.

- `MemoryMax=1G`: Deep parsing loads entire PCAP files into memory for BLE PDU dissection. The C07 design limits file size to 100 MB per file, but concurrent processing of multiple files may consume more memory.
- Requires `@network-io` for potential DB connection (C07 may store enrichment results directly; C08 ML enrichment also needs DB access).

### 4.3 Restart Policy Summary

All services use `Restart=on-failure` with exponential decay handled at the systemd level (`StartLimitBurst`, `StartLimitIntervalSec`). Systemd's built-in rate limiting prevents crash loops from consuming all resources:

| Policy | Value | Meaning |
|--------|-------|---------|
| `Restart` | `on-failure` | Restart only if the process exits with a non-zero exit code or is terminated by a signal. Does not restart if exited with 0 (normal exit) or if stopped by `systemctl stop`. |
| `RestartSec` | `3s` (tshark), `5s` (most), `10s` (ml-classify) | Delay before restart. Short for stateless services (tshark is a shell pipeline), moderate for services with connection pools. |
| `StartLimitBurst` | `3` (most), `5` (heartbeat) | Maximum 3 (or 5) restarts within `StartLimitIntervalSec`. After that, systemd stops trying. |
| `StartLimitIntervalSec` | `120s` | The sliding window for counting burst restarts. 3 restarts in 2 minutes → unit enters `failed` state and stays there. |
| `StartLimitAction` | (default: `none`) | When the start limit is hit, systemd does not perform any automatic action (does not reboot). The operator must manually investigate. |

**Crash loop detection:** If a service exceeds `StartLimitBurst` restarts within `StartLimitIntervalSec`, systemd marks the unit as `failed` and stops attempting restarts. This prevents infinite restart loops from consuming system resources (PF-6: Blast Radius Minimization). The operator is alerted via:
- `systemctl --user status` showing the failed unit
- `tianer_container_failed` Prometheus metric (from C13 Prometheus scraping systemd's D-Bus or textfile collector)
- Grafana dashboard "pipeline-health" panel showing failed units

### 4.5 Dependency Enforcement: Before=/After= Matrix

| Unit | Before | After | Requires |
|------|--------|-------|----------|
| `tianer-net.network` | (all containers) | — | — |
| `tianer-postgres.service` | `tianer-platform.pod`, `tianer-capture.pod`, `tianer-grafana.service` | `tianer-net.network` | `tianer-net.network` |
| `tianer-platform.pod` | — | `tianer-net.network`, `tianer-postgres.service` | `tianer-net.network`, `tianer-postgres.service` |
| `tianer-capture.pod` | — | `tianer-postgres.service` | `tianer-postgres.service` |
| `tianer-grafana.service` | — | `tianer-net.network`, `tianer-postgres.service` | `tianer-net.network`, `tianer-postgres.service` |
| `tianer-tshark@.service` | `tianer-sniffer@.service` | — | — |
| `tianer-ingest@.service` | `tianer-sniffer@.service` | `tianer-tshark@.service`, `tianer-postgres.service` | — |
| `tianer-sniffer@.service` | — | `tianer-tshark@.service` | — |
| `tianer-api.service` | — | `tianer-postgres.service` | — |
| `tianer-gap-detect.service` | — | `tianer-postgres.service` | — |
| `tianer-deep-parse.service` | — | `tianer-rotate.service` | — |
| `tianer-ml-classify.service` | — | `tianer-deep-parse.service`, `tianer-postgres.service` | — |
| `tianer-rotate.service` | — | `tianer-postgres.service` | — |
| `tianer-prometheus.service` | — | `tianer-postgres.service`, `tianer-api.service` | — |
| `tianer-promtail.service` | — | `tianer-postgres.service` | — |
| `tianer-heartbeat@.service` | — | `tianer-sniffer@.service` | — |
| `tianer.target` | — | (all of the above) | — |

**Key ordering properties:**
1. **No circular dependencies.** The graph is a strict DAG. Every `Before=` has a corresponding `After=` on the other side. No pair of units has both `Before=` and `After=` pointing at each other.
2. **PostgreSQL is the central dependency.** All consumers declare `After=tianer-postgres.service`. No consumer declares `Requires=tianer-postgres.service` — this would cause the entire pod to fail if PostgreSQL fails to start. Instead, consumers handle DB unavailability at the application level (exponential backoff reconnect).
3. **tshark → ingest → sniffer ordering (cross-pod via V03 FIFO).** tshark runs in the capture pod, ingest runs in the platform pod — both pods share V03 as a bind-mounted tmpfs. The FIFO reader (ingest) must exist before the sniffer starts writing to prevent the sniffer from blocking on a full pipe buffer at startup.
4. **Pods start in parallel after PostgreSQL.** `tianer-platform.pod` and `tianer-capture.pod` both `After=tianer-postgres.service` but have no ordering between them — they start concurrently.

## 5. Inter-Component Contracts

### 5.1 Container-to-Container Communication Paths

All inter-container communication is explicitly documented. No container implicitly assumes access to another.

| From | To | Transport | Path | Purpose |
|------|-----|-----------|------|---------|
| `tianer-sniffer@` | `tianer-tshark@` | FIFO (V03) | `/var/run/tianer/<name>.fifo` | Raw PCAP stream from sniffer to tshark decoder |
| `tianer-tshark@` | `tianer-ingest@` | FIFO (V03) | `/var/run/tianer/<name>-ingest.fifo` | Structured pipe-delimited fields from tshark to ingest bridge |
| `tianer-ingest@` | `tianer-postgres` | TCP/DNS | `tianer-postgres:5432` on `tianer-net` bridge | COPY protocol bulk insert |
| `tianer-api` | `tianer-postgres` | TCP/DNS | `tianer-postgres:5432` on `tianer-net` bridge | asyncpg parameterized queries |
| `tianer-gap-detect` | `tianer-postgres` | TCP/DNS | `tianer-postgres:5432` on `tianer-net` bridge | Gap detection queries + backfill INSERT |
| `tianer-gap-detect` | PCAP files | Volume mount (V02 :ro) | `/var/lib/tianer/pcap/` | Backfill replay from PCAP archive |
| `tianer-deep-parse` | PCAP files | Volume mount (V02 :ro) | `/var/lib/tianer/pcap/` | Deep BLE protocol analysis from rotated PCAPs |
| `tianer-deep-parse` | JSONL output | Volume mount (V05 :rw) | `/var/lib/tianer/data/` | Write JSONL enrichment records |
| `tianer-ml-classify` | JSONL input | Volume mount (V05 :ro) | `/var/lib/tianer/data/` | Read JSONL from deep parser |
| `tianer-ml-classify` | `tianer-postgres` | TCP/DNS | `tianer-postgres:5432` on `tianer-net` bridge | Write enrichment data to `device_enrichment` |
| `tianer-grafana` | `tianer-postgres` | TCP/DNS | `tianer-postgres:5432` on `tianer-net` bridge | Read-only SQL queries for dashboards |
| `tianer-prometheus` | `tianer-api` | TCP/HTTP | `tianer-api:8080/metrics` on `tianer-net` bridge | Scrape Python service metrics |
| `tianer-prometheus` | Metrics textfiles | Volume mount (:ro) | `/var/lib/tianer/metrics/` | Read file-based metrics from C++ services |
| `tianer-rotate` | PCAP files | Volume mount (V02 :rw) | `/var/lib/tianer/pcap/` | Rename, compress, delete rotated PCAP files |
| `tianer-heartbeat@` | Heartbeat files | Volume mount (:rw) | `/var/lib/tianer/heartbeat/<name>.ts` | Write sniffer heartbeat timestamps |

**Communication not listed above is prohibited.** Specifically:
- No container may access another container's process namespace.
- No container may bind to a port exposed by another container without an explicit contract above.
- Platform pod containers share V03 (FIFOs) with the capture pod via host tmpfs bind mount — this is the cross-pod FIFO IPC mechanism.
- Platform pod containers cannot access USB devices — those are exclusive to the capture pod.

### 5.2 Contract ORCHESTRATION-1: systemd Unit Dependency

**From:** C12 (Service Orchestration)
**To:** C14 (Deployment Automation)
**Format:** Quadlet `.container` files with correct `[Unit]` dependency directives

C14's `setup.sh` Phase 7 deploys Quadlet files; `systemctl daemon-reload` generates systemd `--user` units. C14's Phase 11 enables specific service instances. C12 specifies the dependency graph that Quadlet's `[Unit]` sections must encode. If a service's Quadlet file does not declare the correct `After=` on its dependency, the startup ordering is undefined and the pipeline may fail with connection-refused errors.

**Verification:** `systemd-analyze --user dot tianer.target` produces a dependency graph. If any edge is missing, the Quadlet file is incorrect.

## 6. Failure Modes & Recovery

### 6.1 Failure Catalog

| # | Failure Mode | Detection | Impact | Recovery |
|---|-------------|-----------|--------|----------|
| **F-C12-1** | **Circular dependency** | `systemctl --user daemon-reload` fails with "ordering cycle" error. `systemd-analyze --user verify` reports the cycle. | All affected units fail to load. Platform cannot start. | Fix the Quadlet `[Unit]` directives to break the cycle. Run `systemd-analyze verify` locally before deployment. Quadlet does not allow self-referencing `Before=`/`After=`, but cycles can be created across multiple files (A after B, B after C, C after A). |
| **F-C12-2** | **Service crash loop** | `systemctl --user status <unit>` shows repeated `activating → failed → restarting` or `StartLimitBurst` hit. | That component is unavailable. Restarts consume CPU. After `StartLimitBurst` (3 restarts in 120s), systemd stops trying — the unit enters `failed` state. | Operator investigates `podman logs <container>` for the root cause. Fix and run `systemctl --user reset-failed <unit>` to reset the start limit counter, then `systemctl --user start <unit>`. |
| **F-C12-3** | **Ordering violation — service starts before dependency ready** | Application-level connection errors (e.g., ingest bridge gets `ECONNREFUSED` to PostgreSQL). | Service fails to initialize. If `Restart=on-failure`, it restarts and succeeds on retry (dependency becomes ready during the restart delay). | Application-level exponential backoff reconnect handles this (C02 §6.2). systemd `After=` only orders the startup attempt, not readiness. Readiness is the application's responsibility. |
| **F-C12-4** | **Pod startup failure — container in pod exits** | `podman pod start` fails. systemd generates `pod-*.service` unit that enters `failed` state. Whole pod fails to start. | All containers in the pod are unavailable. Capture pipeline or platform services down. | `podman logs <failed-container>` for root cause. Common causes: missing volume mount, image pull failure, environment variable missing. Fix and restart the pod: `systemctl --user start tianer-capture-pod.service`. |
| **F-C12-5** | **Podman daemon failure** | `podman system info` fails. `systemctl --user start tianer-postgres.service` fails with "Cannot connect to Podman." | All containers are unavailable. Complete platform outage. | Rootless Podman is managed by systemd `--user`. Restart the user session: `systemctl --user restart podman.socket`. If the user session itself has failed: `loginctl enable-linger tianer` and restart the session. Host reboot is the nuclear option (C01 recovery sequence). |
| **F-C12-6** | **systemd --user failure** | `systemctl --user status` returns "Failed to connect to bus." `loginctl show-user tianer -p Linger` returns `no`. | All platform services stop. systemd-linger daemon may have crashed or been disabled. | Re-enable linger: `sudo loginctl enable-linger tianer`. This restarts the `systemd --user` instance. All services resume from their last state. |
| **F-C12-7** | **Timer missed — oneshot not triggered** | `systemctl --user list-timers` shows timer `last` elapsed > expected interval. | PCAP rotation not occurring → V02 fills up. Deep parser not running → V05 goes stale. | Check `Persistent=true` is set on all timer units. Manually run `systemctl --user start tianer-rotate.service` to catch up. If the timer unit has `Persistent=false`, missed runs are not caught up — convert to `Persistent=true`. |
| **F-C12-8** | **Oneshot service hangs** | `systemctl --user status tianer-deep-parse.service` shows `activating` for longer than `TimeoutStartSec`. | Deep parser stuck (e.g., corrupted PCAP file causes infinite loop). Other services unaffected — oneshot runs independently. | systemd kills the oneshot after `TimeoutStartSec` (3600s for deep parser). The unit enters `failed` state. Operator investigates the PCAP file that caused the stall. Manually restart: `systemctl --user start tianer-deep-parse.service`. |
| **F-C12-9** | **Volume mount missing or wrong mode** | Container fails to start with "permission denied" or "no such file or directory" for a mount path. | That container fails. Other containers may be unaffected if their volumes are correct. | Verify V01–V09 host paths exist and have correct ownership (`tianer:tianer`). Verify Quadlet `Volume=` directive matches storage-strategy.md access matrix. Verify the access mode (`:ro` or `:rw`) is correct per the per-container specification in §4.1. |
| **F-C12-10** | **Image pull failure** | `podman pull localhost/tianer-ingest` fails. Container stays in `created` state, never enters `running`. | That component cannot start. | Check image exists: `podman images localhost/tianer-ingest`. Rebuild: `make build-ingest`. Check digest pinning in Quadlet `Image=` directive — if digest is stale (image rebuilt with new digest), update the Quadlet file or remove the digest pin for local images. |
| **F-C12-11** | **Resource limit exceeded (OOM)** | Container is killed by cgroups (OOM killer) [4]. `podman inspect` shows `OOMKilled: true`. | That container restarts. Buffer data may be lost (ingest batch). PCAP data is safe on V02. | Increase `MemoryMax` if the limit is too restrictive. For ingest bridge, 512 MB covers 300K row buffer (~60 MB) with generous headroom. For deep parser, 1 GB covers 100 MB PCAP files with processing overhead. If OOM persists at these limits, investigate memory leak in the application. |
| **F-C12-12** | **sniffers.yaml parsing error during template instantiation** | `setup.sh` Phase 11 fails to parse `sniffers.yaml`. Template units not enabled. | Capture pod starts with no sniffer instances — no capture occurs. API, gap detector, and other services remain functional (graceful degradation per PF-4). | Fix `sniffers.yaml` syntax. Re-run `setup.sh` Phase 11 to enable sniffer units. |

### 6.2 Crash-Only Design Compliance (PF-8)

Every systemd unit is designed for crash-only operation:

| Unit | Safe to kill at any time? | Recovery on restart |
|------|--------------------------|---------------------|
| `tianer-postgres.service` | Yes — `kill -9` triggers WAL recovery on restart | PostgreSQL WAL restores consistency within 5 seconds |
| `tianer-sniffer@.service` | Yes | Reconnects USB device, resumes capture. Heartbeat gap detected and backfilled by C06. |
| `tianer-tshark@.service` | Yes | tshark exits, PID file cleaned by systemd, FIFO persists (tmpfs). Restart re-opens FIFO. |
| `tianer-ingest@.service` | Yes | In-memory batch buffer lost. Gap detector identifies the missing window and backfills from PCAP (V02). Zero PCAP data loss. |
| `tianer-api.service` | Yes | Connection pools recreated on restart. In-flight requests may fail — frontend retries. |
| `tianer-gap-detect.service` | Yes | Gap detection cycle resumes. Idempotent — gaps detected on next scan independent of restart timing. |
| `tianer-deep-parse.service` (oneshot) | Yes | Mid-processing PCAP may have partial output. Next run processes the same file again (idempotent by filename tracking). |
| `tianer-grafana.service` | Yes | SQLite DB in V07 may have dirty pages; Grafana recovers on restart. No data loss. |

**No unit requires a "clean shutdown" for data integrity.** If a unit does require clean shutdown, the design is wrong (PF-8). The PCAP files (V02) are the source of truth — all other data is recoverable from them.

### 6.3 Recovery Service Dependency

| Recovery Agent | Failure It Handles | Available After Reboot? |
|---------------|-------------------|------------------------|
| systemd `--user` (linger) | Container crash (F-C12-2,4) | Yes — starts automatically via `loginctl enable-linger` |
| systemd `Restart=on-failure` | Service crash loop (F-C12-2) | Yes — built-in systemd feature |
| `StartLimitBurst` rate limiting | Crash loop protection (F-C12-2) | Yes — built-in systemd feature |
| Application reconnect (C02 §6.2) | DB unavailability (F-C12-3) | Yes — per-component behavior |
| `Persistent=true` timer catch-up | Missed timer runs (F-C12-7) | Yes — built-in systemd timer feature |
| C06 Gap Detector | Missed ingest buckets (F-C12-2,11) | Yes — auto-starts, backfills from PCAP |
| C04 PCAP Rotation | Disk fill (F-C12-7) | Yes — timer-triggered |
| Operator `systemctl reset-failed` | Permanent failure after StartLimitBurst | Manual intervention |

| `tianer-ingest@` | `/var/run/tianer`, `/var/log/tianer`, `/tmp` |
| `tianer-heartbeat@` | `/var/lib/tianer/heartbeat`, `/var/log/tianer`, `/tmp` |
| `tianer-gap-detect` | `/var/log/tianer`, `/tmp` |
| `tianer-deep-parse` | `/var/lib/tianer/data`, `/var/log/tianer`, `/tmp` |
| `tianer-ml-classify` | `/var/log/tianer`, `/tmp` |
| `tianer-rotate` | `/var/lib/tianer/pcap`, `/var/log/tianer`, `/tmp` |
| `tianer-prometheus` | `/var/lib/prometheus`, `/tmp` |
| `tianer-promtail` | `/tmp` |

**Cross-volume access enforcement:** The `ReadWritePaths` allowlist is the **enforcement mechanism** for the access matrix defined in storage-strategy.md. A container that mounts V02 as a volume but does not include `/var/lib/tianer/pcap` in `ReadWritePaths` can only access the volume in read-only mode (because Podman's default for bind mounts without explicit `:rw` is read-only, and `ReadWritePaths` further restricts even `:rw` mounts at the systemd level). This is defense in depth (PF-7).

### 8.2 NoNewPrivileges Enforcement

`NoNewPrivileges=true` is set on **every container**. This kernel feature prevents:
- Gaining new capabilities via `execve()` of a setuid binary
- Transitioning to a new security context via SELinux/AppArmor
- Modifying the `no_new_privs` bit after it's set

With `NoNewPrivileges=true`, even if an attacker compromises a container process and executes a setuid binary inside the container, the new process does not gain elevated privileges. This is a critical blast-radius minimization control (PF-6).

### 8.3 --cap-drop ALL Default

All containers start with `DropCapability=ALL` and only receive explicit `AddCapability` directives:

| Container(s) | Capabilities Added | Justification |
|-------------|-------------------|---------------|
| `tianer-postgres` | `CHOWN DAC_OVERRIDE FOWNER SETGID SETUID` | PostgreSQL needs to change file ownership and override discretionary access controls for its data directory |
| `tianer-sniffer@`, `tianer-tshark@` | `NET_RAW NET_ADMIN` | `dumpcap` requires raw packet capture capabilities (C01 §4.6.1) |
| All others | (none) | All other services operate entirely at the application layer — DB queries over TCP, HTTP serving, file I/O on mounted volumes. No kernel capabilities needed. |

### 8.4 No --privileged Containers

Zero Quadlet files contain `Privileged=true`. Zero `.container` files use `--privileged` in their podman arguments. This is a hard requirement per the container security recommendation. Every privilege escalation path is auditable through the `DropCapability`/`AddCapability` directives, not through a blanket "everything allowed."

### 8.5 User= Directive

All containers run as `User=tianer` inside the container. This maps to the host UID of the `tianer` user (system UID, typically < 1000) via the user namespace mapping configured by rootless Podman. The container's root (UID 0) maps to the host's `tianer` UID, so even if a process inside the container runs as UID 0, it only has the host privileges of the `tianer` user — a non-sudo, no-login-shell system user.

### 8.6 EnvironmentFile Injection Security

Secrets are injected via `EnvironmentFile` directives pointing to host files:

| Secret File | Consumer Containers |
|-------------|-------------------|
| `/etc/tianer/secrets/db_password.env` | `tianer-postgres`, `tianer-ingest@`, `tianer-api`, `tianer-gap-detect`, `tianer-ml-classify` |
| `/etc/tianer/secrets/ro_db_password.env` | `tianer-api` (read pool) |
| `/etc/tianer/secrets/writer_db_password.env` | `tianer-ingest@`, `tianer-gap-detect`, `tianer-ml-classify` |
| `/etc/tianer/secrets/grafana_db_password.env` | `tianer-grafana` |
| `/etc/tianer/secrets/api_key.env` | `tianer-api` |

These EnvironmentFiles are only readable by the `tianer` user on the host (mode 0600). The secrets are **never baked into container images** — they are injected at runtime by Podman when the Quadlet-generated systemd unit starts the container.

### 8.7 SystemCallFilter Coverage

Every container uses `SystemCallFilter` with an explicit allowlist. No container uses the blank `SystemCallFilter=` (which would allow all syscalls). See §4.4 for the per-group breakdown. The union of allowed syscalls is the intersection of `@system-service`, `@file-system`, `@process`, `@signal`, and (where network is needed) `@network-io` — approximately 150–180 syscalls out of ~430 available on Linux. This blocks:
- Kernel module loading (`init_module`, `finit_module`, `delete_module`)
- Raw disk I/O (`iopl`, `ioperm`)
- System calls that modify kernel state (`kexec_load`, `reboot`)
- Obsolete syscalls (`create_module`, `get_kernel_syms`, `uselib`)

**Verification:** `systemd-analyze syscall-filter` lists groups. `systemd-analyze --user security tianer-postgres.service` provides a security exposure score (0–10, lower is safer).

### 9.3 Resource Limit Tuning

All `MemoryMax` and `CPUQuota` values in §4.1 are **conservative defaults** for the Raspberry Pi CM5 (8 GB RAM, 4× Cortex-A76 cores). Operators can tune these per `/etc/tianer/tianer.env` overrides:

```bash
# Optional per-service resource overrides (uncomment to enable)
# TIANER_POSTGRES_MEMORY_MAX=2G
# TIANER_INGEST_MEMORY_MAX=256M
# TIANER_API_MEMORY_MAX=256M
```

These overrides are read by C14's `setup.sh` Phase 11, which replaces the `${...}` placeholders in the deployed Quadlet files. If no override is set, the hardcoded defaults in the Quadlet files apply.

### 9.4 Sniffer Instance Configuration

The number of active sniffer instances is driven by `/etc/tianer/sniffers.yaml`. C14 Phase 11 reads this file to determine which template units to enable. C12 does not hardcode the instance count — it is dynamic based on the YAML configuration:

```yaml
# Example: 2 sniffers enabled (ut1 + nrf1), 2 disabled (nrf2, nrf3)
sniffers:
  - name: ut1      enabled: true   → tianer-sniffer@ut1.service ENABLED
  - name: nrf1     enabled: true   → tianer-sniffer@nrf1.service ENABLED
  - name: nrf2     enabled: false  → NOT enabled
  - name: nrf3     enabled: false  → NOT enabled
```

Changing `enabled: true` to `enabled: false` and restarting the capture pod reduces the instance count. Changing `false` to `true` and running `make install` (to re-enable the units) adds new instances.

### 9.5 Target Dependencies

The `tianer.target` unit's `Wants=` list is the definitive list of all top-level services. Adding a new service requires:
1. Creating its Quadlet `.container` file
2. Adding `Wants=<new-service>.service` to `tianer.target`
3. Running `make install` (C14) and `daemon-reload`

Removing a service is the reverse.

## 11. Deployment Notes

### 11.1 Quadlet File Deployment

Quadlet files are deployed from the repository source (`deploy/containers/`) to the Quadlet system directory (`/etc/containers/systemd/`) by C14 `setup.sh` Phase 7:

```bash
# Deploy all Quadlet files (C14 Phase 7)
install -m 0644 deploy/containers/*.container /etc/containers/systemd/
install -m 0644 deploy/containers/*.pod       /etc/containers/systemd/
install -m 0644 deploy/containers/*.volume     /etc/containers/systemd/
install -m 0644 deploy/containers/*.network    /etc/containers/systemd/
systemctl daemon-reload
```

This must be followed by enabling user services and starting them:

```bash
# As the tianer user (or via machinectl)
systemctl --user daemon-reload
systemctl --user enable tianer-postgres.service
systemctl --user enable tianer-grafana.service
systemctl --user enable tianer-capture-pod.service
systemctl --user enable tianer-platform-pod.service
systemctl --user start tianer.target
```

### 11.2 systemd-linger Prerequisite

Rootless Podman with Quadlet requires `loginctl enable-linger tianer` (performed by C01 Phase 6). Without lingering, the `systemd --user` instance for the `tianer` user terminates when the last session ends — containers would stop on logout. Lingering ensures the user instance starts at boot and persists indefinitely.

Verification:
```bash
loginctl show-user tianer -p Linger
# Expected: Linger=yes
```

### 11.3 Operator Commands

```bash
# Start/stop the entire platform
systemctl --user start tianer.target
systemctl --user stop tianer.target
systemctl --user restart tianer.target

# Start/stop individual pods
systemctl --user start tianer-capture-pod.service
systemctl --user stop tianer-platform-pod.service

# Start/stop individual containers
systemctl --user start tianer-api.service
systemctl --user stop tianer-sniffer@ut1.service

# Check status
systemctl --user status tianer-*

podman logs tianer-api

systemctl --user reset-failed tianer-api.service
systemctl --user start tianer-api.service

# Manually trigger oneshot runs
systemctl --user start tianer-rotate.service
systemctl --user start tianer-deep-parse.service

# Check timer status
systemctl --user list-timers
```
